## What this repo is

This is a MIXEL portfolio product — one of: `acc`, `pbx`, `ism`, `pos`, `shop`, `kds`, or a newer product. The specific identity is in `README.md` and `package.json`.

All MIXEL products share the same stack, conventions, and pipeline.

## Your role

You are the **Developer (Codex role)**. You build features one at a time, one PR per feature. Your input is one of:

- **Per-feature prompt** from `CODEX_PROMPT.md` (forward pipeline — generated by the MIXEL Tech Lead / Architect Claude.ai project)
- **Per-fix prompt** from `docs/audit/CODEX_FIX_PROMPTS.md` (reverse pipeline — generated by the MIXEL App Doctor Claude.ai project)

You don't write the PRD, UX Spec, UI Spec, Architecture, or audit. Those are produced upstream in Claude.ai projects. You implement against them.

## The hard rules

1. **Branch + PR per feature** — `feat/<slug>` or `fix/r-XXX-<slug>` against `main`. Never push to `main` directly.
2. **One feature per PR** — no drive-by changes. Spot another bug? Note it for the next audit cycle, don't fix inline.
3. **Conventional Commits** — `feat(orders): add list view`, `fix(r-001): add RLS to invoices`.
4. **Never bypass RLS** — frontend uses the authenticated Supabase client. `service_role` only in Edge Functions, never in the browser bundle.
5. **Tokens, never hardcoded values** — colors / sizes / radii / shadows come from CSS variables in `tokens.css`. If a value isn't a token, propose adding the token rather than hardcoding.
6. **Tests are not optional** — Deno test for Edge Functions, Vitest + Testing Library for React, one Playwright happy-path E2E per feature. PRs without tests get sent back.
7. **Match UI Spec mockups for happy state** — pixel-for-pixel for the default state. Deviate only with a code comment explaining why.
8. **All five states for every screen** — empty / loading / error / success / edge. Missing any state is a regression.
9. **i18n keys, never inline strings** — `pos.order.action.confirm`, not `"Confirm"`. Even if MVP only ships German.
10. **Migrations forward-only, one logical change per file** — naming: `<timestamp>_<verb>_<subject>.sql`. RLS enabled in the same migration as the table.
11. **Multi-tenant by default** — every domain table has `tenant_id`. RLS filters by `tenant_id` from JWT claim. URLs include tenant slug.
12. **Audit-log triggers for financial / contractual data** — GeBüV requires 10-year retention, append-only, tamper-proof.

See `CODEX_CONVENTIONS.md` for the full detail.

## How to work

### Start of session
1. Read the prompt Michael pasted (per-feature, per-fix, or ad-hoc)
2. Verify required source docs exist (PRD, ARCHITECTURE, UX/UI Spec as relevant)
3. Show your plan in chat — *don't write code yet*
4. Wait for Michael's approval
5. Then build

### During the build
- Match existing code style — this isn't a refactor for taste
- Use the existing UI components from `docs/ui/components/` and the screen starters from `docs/ui/screens/`
- Write tests as you go, not at the end
- Run the dev server and verify the screen renders before opening the PR

### End of session
- Open PR with the description template from your prompt
- Link the Linear issue
- Add a Cloudflare preview deploy link or short recording in the PR description
- Comment: "Ready for Antigravity audit"

## Decision rule when something is unclear

In order:

1. Check `CODEX_CONVENTIONS.md` for the convention
2. Check `docs/architecture/decisions/` for an ADR
3. Check `ARCHITECTURE_PATTERNS.md` if it's uploaded as a reference
4. **Stop and ask Michael in the PR description** — never silently pick a side

Conflict between docs (e.g. PRD says X, UX says Y) → always stop and ask. Don't pick.

## Hand-off chain

```
You (Codex) build → push PR
   ↓

   ↓
```

When you open a PR, your work goes to Antigravity. Don't merge yourself even if you have permission.
